# Minimal Test World Creation

When tests need a specific room layout (e.g. a BOSS room at known coordinates), prefer building a minimal world with `createRoom()` over calling `generateWorldMap()`. This avoids flaky world generation, pre-existing generator integration failures, and makes coordinate geometry directly verifiable.

## Pattern

```javascript
// Build a 2×2 world with one normal room and one boss room
const world = createWorld(2, 2);

// Room (0,0): normal room
world.rooms[0][0] = createRoom(ROOM_TYPE.NORMAL, 0, 0);

// Room (1,0): boss room at known coordinates
world.rooms[0][1] = createRoom(ROOM_TYPE.BOSS, 1, 0);

// Verify the room is where we expect
const room = getRoomAt(world, 1, 0);
expect(room.type).toBe(ROOM_TYPE.BOSS);
```

## When to Use

- **Focused geometry tests** — e.g. "snake at (25, 25) moves RIGHT to (26, 25)" needs a known tile layout, not random generation
- **Multi-room transitions** — e.g. testing snake walking from room (0,0) into room (1,0) through a door
- **Boss room interactions** — e.g. testing boss intro dismissal with a specific room arrangement
- **Null room testing** — create a 2×2 grid where one room slot is `null` to test boundary behavior

## Pitfalls

- `createRoom()` creates a room with default coordinates — ensure `roomX`/`roomY` match the grid position
- Border walls are generated by default — don't assume FLOOR at column 0 or row 0
- `getRoomAt(world, rx, ry)` uses `world.rooms[ry][rx]` (y first!) — double-check coordinate order
- After modifying room tiles, verify the cell is what you expect with `getCellAt(world, worldX, worldY)`
