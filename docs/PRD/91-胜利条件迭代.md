# Research: 胜利条件迭代 — Boss 房间战斗

> Parent Issue: #91
> Agent: research-agent
> Date: 2026-07-09

---

## 1. Problem Definition

### Current Behavior

当前游戏胜利条件仅需蛇头进入 GOAL 类型房间：

```js
// core.js tick() — 进入 GOAL 房间时立即胜利
if (newRoom.type === ROOM_TYPE.GOAL) {
  s.gameState = 'won';
  return s;
}
```

- GOAL 房间只是一个普通房间，带有 `★ GOAL` 标记
- 玩家进入后没有任何战斗或挑战，立即胜利
- 游戏流程简单：探索 → 收集钥匙 → 开门 → 进入 GOAL 房间 → 通关
- 胜利画面直接展示，没有 BOSS 战斗环节

### Expected Behavior

根据 Issue #91 描述，胜利条件应迭代为：

1. **玩家到达 GOAL 房间后**，房间入口关闭（蛇穿过的门变为不可通行）
2. **BOSS 战触发**：房间内出现 BOSS（第一关 BOSS 为蓝色蛇），击败后才是真正胜利
3. **BOSS 攻击机制**：BOSS 会吐子弹攻击玩家，每次吐子弹自身长度减一
4. **玩家被 BOSS 子弹击中**：玩家身体长度减一，断掉的身体块会遵循物理「弹飞」，成为可拾取的「食物」
5. **玩家攻击 BOSS**：子弹击中 BOSS 后，BOSS 身体断掉一块，变为「食物」
6. **BOSS 吃食物**：BOSS 长度增加一（如同玩家吃食物一样增长）
7. **食物自动生成**：房间任意位置，每隔一段时间会生成一个食物

### User Scenarios

- **Scenario A（初次 BOSS 战）：** 玩家费尽千辛万苦拿到钥匙到达 GOAL 房间 → 房间入口关闭 → BOSS 出现 → 战斗开始。胜利的成就感大幅提升。
- **Scenario B（BOSS 战策略）：** 玩家需要合理分配攻击频率（消耗自身长度发射子弹）和吃食物（恢复长度）之间的平衡，同时躲避 BOSS 的子弹。
- **Scenario C（多周目/多关卡）：** 不同关卡的不同 BOSS（蓝色蛇为第一关），未来可扩展更多 BOSS 类型。
- **Frequency:** 每局游戏一次 BOSS 战，但 BOSS 战长度决定了通关所需的核心操作量。

## 2. Root Cause Analysis (Bug) / Design Intent (Feature)

### Why Does Current Behavior Exist?

#15 银河城重构时将胜利条件定义为「到达 GOAL 房间即胜利」。这是一个 MVP 设计决策，为后续扩展保留空间：
- GOAL 房间已作为特殊房间类型存在
- 房间数据结构支持添加 BOSS 实体
- 核心战斗系统（子弹、敌人碰撞、食物系统）已就绪
- 目的是先让完整游戏循环跑通，再迭代 BOSS 战

### Why Change Now?

1. **游戏高潮缺失**：GOAL 房间通行即胜利，缺乏最终挑战，游戏体验虎头蛇尾
2. **战斗系统未充分利用**：已有的子弹、碰撞、长度增减系统在普通敌人上体现不足，需要 BOSS 这种高强度战斗场景来验证
3. **可扩展性**：BOSS 战斗是游戏内容扩展的基石——不同 BOSS 对应不同关卡，这是游戏从「demo」走向「正式版」的关键一步
4. **「弹飞」物理机制**：身体断块遵循物理飞出的设定，引入了新的、有趣的游戏机制，增加了动态性

### Previous Constraints

- **零外部依赖**：物理弹飞需要纯 JS 模拟（无需物理引擎）
- **保持单 HTML 文件架构**：所有渲染和逻辑在现有 `public/` 目录下
- **模块化引擎结构**：新增 BOSS 逻辑放在 `public/src/engine/` 相应模块
- **现有 GOAL 房间数据结构不变**：在房间上叠加 BOSS 数据而非重构
- **GameBoy 调色板约束**：BOSS 需要与现有敌人视觉区分

### Data Flow Impact

```
[Before — GOAL room = instant win]
tick() → roomTransition → newRoom.type === GOAL → gameState = 'won'

[After — GOAL room = boss fight]
tick() → roomTransition → newRoom.type === GOAL →
  └─→ 第一次进入: door closes, createBoss, bossFightActive = true
  └─→ BOSS 战斗中:
          ├─ 蛇移动 + 碰撞检测（墙、食物、BOSS、BOSS子弹）
          ├─ BOSS AI: 移动/射击决策
          ├─ BOSS 子弹：发射 → 飞行 → 碰撞玩家/墙
          ├─ 玩家子弹：发射 → 飞行 → 碰撞BOSS/墙
          ├─ 碰撞结果：
          │   ├─ 玩家被BOSS子弹击中 → 长度-1 → 断裂块弹飞 → 变为食物
          │   ├─ 玩家子弹击中BOSS → BOSS长度-1 → 断裂块弹飞 → 变为食物
          │   └─ 玩家撞击BOSS身体 → 长度-1
          ├─ BOSS吃食物 → 长度+1
          ├─ 食物计时 → 定时生成食物
          └─ BOSS长度归零 → gameState = 'won'
```

## 4. Solution Comparison

### Approach A: 在现有 GOAL 房间基础上叠加 BOSS 战（推荐）

**Description:** 改造 GOAL 房间的处理逻辑：
- 维护一个 `bossFightActive` 状态标记
- 蛇头第一次进入 GOAL 房间时：关闭入口门、实例化 BOSS 实体、设置 `bossFightActive = true`
- BOSS 战期间：tick() 调用 BOSS AI 模块（移动、射击、进食）、处理玩家/BOSS 子弹碰撞
- BOSS 子弹与玩家子弹使用同一 projectile 数据结构，但添加 `source:'boss' | 'player'` 标记
- 弹飞食物是一个新的实体类型：`FlyingFood { x, y, vx, vy, lifetime, isFood: true }`
- BOSS 长度归零 → 设置 `gameState = 'won'`

**BOSS 设计细节：**

```
Boss {
  id,
  x, y,               // 世界坐标
  segments: [{x,y}],  // 身体段
  hp,                 // 等于长度
  length,             // 当前长度
  type: 'blue_snake',
  speedTicks,          // 移动间隔
  tickCounter,         // 计数
  shootCooldown,        // 射击冷却计数
  shootInterval,       // 射击间隔（tick数）
  aiState: 'idle' | 'chase' | 'retreat',
}
```

**数据结构新增：**

```js
// GameState 新增字段
{
  bossFightActive: false,    // BOSS 战是否激活
  boss: null,                // BOSS 实体
  roomEntranceClosed: false, // GOAL 房间入口是否关闭
  flyingFood: [],            // 弹飞的食物块
  bossFoodTimer: 0,          // BOSS 房间食物生成计时器
}
```

**入口关闭实现：** 当蛇头进入 GOAL 房间后，在 `world.rooms[goalY][goalX].doors` 中标记所有门为关闭状态（设置 blocked 为 true），或修改门对应的 CELL 为 WALL。

**食物弹飞物理（纯 JS 模拟）：**
```js
FlyingFood {
  x, y,                   // 当前世界坐标
  vx, vy,                 // 速度向量（方向随机 + 垂直于碰撞方向）
  lifetime: 60,           // tick 数后消失
  bounceCount: 0,         // 弹跳次数
  angular: 0,             // 旋转角度
  angularSpeed: rand,     // 旋转速度
  isFood: true,           // 可作为食物被蛇/BOSS 吃掉
  settled: false,         // 是否落定（变为普通食物）
}
```

tick 更新：
```
flyingFood.x += vx
flyingFood.y += vy
vx *= 0.95, vy *= 0.95  // 摩擦力衰减
lifetime--
if 碰撞墙壁 → 反弹 (vx *= -0.5)
if lifetime <= 0 → 落定 → 变为 room.entities.food[] 中的普通食物
```

**BOSS 射击 AI：**
```
boss.shootCooldown 每 tick -1
if boss.shootCooldown <= 0:
  // 瞄准玩家方向
  dir = 计算从 BOSS 头部到玩家头部的方向（取最近的轴向）
  createBossBullet(boss.x, boss.y, dir)
  boss.length -= 1
  boss.segments.pop()
  boss.shootCooldown = boss.shootInterval
  // 同时生成一个 FlyingFood（BOSS 断掉的那块飞到随机方向）
  createFlyingFood(boss.x, boss.y, randomDir)
```

**Pros:**
- 对现有 GOAL 房间基础设施改动最小（复用房间、门、坐标系统）
- BOSS 实体可以复用现有 enemy 的数据结构（`segments[]`、`hp`）并扩展
- BOSS 子弹复用现有 `projectile` 系统，仅添加 `source` 标记
- 入口关闭可通过修改 `world` 中的 door 属性实现，简单直接
- 弹飞食物独立管理系统，不影响现有食物逻辑

**Cons:**
- `tick()` 函数复杂度增加（需要额外分支处理 BOSS 战）
- BOSS AI 需要新模块
- 弹飞物理需要新实体的渲染支持

**Risk:** Low-Medium — 架构清晰，与现有系统解耦良好
**Effort:** Medium（预计 6-10 小时）

### Approach B: 引入独立的 BOSS 房间类型（BOSS_ROOM）

**Description:** 新增 `ROOM_TYPE.BOSS`，在 GOAL 房间之前的必经路径上放置一个独立的 BOSS 房间：

- 胜利条件保持为「进入 GOAL 房间即胜利」
- 但在 GOAL 房间前必须通过一个 BOSS 房间
- 地图生成器在 GOAL 房间相邻位置强制生成一个 BOSS 房间
- BOSS 房间内与 BOSS 战斗，击败 BOSS 后通往 GOAL 房间的门打开

**Pros:**
- 不修改现有胜利逻辑（纯扩展）
- BOSS 房间逻辑独立，可单独维护
- 未来可添加多个 BOSS 房间（多关卡）

**Cons:**
- 地图需要额外一个房间，地图规模需从 5×5 扩展到 6×6
- 地图生成器需要处理 BOSS 房间类型分配和连通性保证
- 维护两套胜利条件（GOAL 房间 + BOSS 房间前的通关判定）导致逻辑分裂
- BOSS 房间到 GOAL 房间的过渡设计复杂（是直接相连接廊还是什么？）
- 与 Issue #91 描述「到达该房间后，房间入口关闭，要击败一个boss才能胜利」不完全一致
- 用户体验上多了一个额外的"步骤"，不流畅

**Risk:** Medium — 地图生成器变更影响全局
**Effort:** Medium（预计 6-8 小时）

### Approach C: BOSS 作为独立的顶级战场（新场景）

**Description:** 进入 GOAL 房间后，游戏切换到一个独立的 BOSS 战斗场景（新 Canvas 渲染、新游戏循环）：

- 进入 GOAL 房间 → 触发场景切换 → 渲染 BOSS 战斗独立界面
- BOSS 战斗场景中只有玩家蛇和 BOSS，背景为纯色竞技场（无 tiles）
- 战斗结束后，切换回主游戏渲染胜利画面

**Pros:**
- 与主游戏引擎完全解耦
- BOSS 战场可以不限于 20×20 网格
- 视觉上更震撼（全屏战场）

**Cons:**
- 需要维护两套渲染系统（主场景 + BOSS 战场）
- 状态在两个场景间传递复杂（蛇的长度、分数如何同步？）
- 数据流动复杂化
- 工程量翻倍
- 与现有 GameBoy 风格不一致（场景切换突兀）
- 玩家丢失房间上下文（房间里的食物、门都不在了）

**Risk:** High — 本质上是一个新游戏，集成风险大
**Effort:** Large（预计 12-20 小时）

### Recommendation

→ **Approach A** 因为：

1. **最小改动**：复用现有 GOAL 房间、门系统、碰撞系统、projectile 系统
2. **自然延续**：蛇进入 GOAL 房间 → 入口关闭 → BOSS 出现 → 在同一个空间战斗
3. **易于理解**：玩家的蛇和 BOSS 蛇在同一网格中对战，符合贪吃蛇的直觉
4. **扩展性强**：未来可以添加更多 BOSS 类型（`boss.type` 枚举），不同 BOSS 有不同的射击模式、速度、长度
5. **弹飞食物机制**丰富了战斗动态——它不仅是一个惩罚（被打到掉长度），也是机会（可以吃到弹飞的食物）
6. **与 Issue #91 描述完全吻合**：「到达该房间后，房间入口关闭，要击败一个boss才能胜利」

## 5. Boundary Conditions & Acceptance Criteria

### Normal Path

1. 玩家探索地图收集钥匙和食物 → 到达 GOAL 房间入口
2. 蛇头从门口进入 GOAL 房间 → 入口在所有方向关闭（`doors` 全部设置 locked 或 blocked）
3. BOSS（蓝色蛇）在房间中心出现 → `bossFightActive = true`
4. BOSS 初始化长度 = 10（可调），比普通敌人更长
5. Boss 开始追逐玩家 + 每隔 N 次 tick 发射子弹
6. 玩家躲避 BOSS 子弹，同时发射子弹反击
7. BOSS 子弹击中玩家 → 玩家长度-1 → 玩家断掉的块弹飞 → 变为 FlyingFood
8. 玩家子弹击中 BOSS → BOSS 长度-1 → BOSS 断掉的块弹飞 → 变为 FlyingFood
9. BOSS 经过食物或 FlyingFood → BOSS 吃掉并长度+1
10. 玩家吃到食物/FlyingFood → 玩家长度+1
11. 房间内每隔 M 个 tick 自动生成一个食物（随机位置，不在蛇/BOSS/障碍物上）
12. BOSS 长度归零 → BOSS 击败 → `gameState = 'won'` → 胜利画面
13. 如果玩家长度归零 → `gameState = 'gameover'` → 加载存档点

### Edge Cases

1. **BOSS 初始化时房间无空位：** GOAL 房间被障碍物占满 → BOSS 生成前应确保至少有 20 个空位。地图生成器为 GOAL 房间预留足够空间（清除所有非地板、非门的 CELL）。
2. **蛇进入 GOAL 房间时长度很短（< 3）：** BOSS 战对短蛇极不公平 → 方案一：禁止进入（长度锁，进入 GOAL 前需要最小长度，例如 5）；方案二：允许进入但战斗更困难（短蛇速度更快，作为平衡）。
3. **BOSS 连续射击导致长度迅速归零：** Boss 过度射击导致自身长度降至 0 而失败 → 设置 shootCooldown 最小值为 3 tick，防止 BOSS 自杀。
4. **弹飞食物飞出房间边界：** FlyingFood 到达房间边界时应反弹（模拟物理碰撞）或停留在边界格子上变成普通食物。
5. **FlyingFood 飞行到蛇/BOSS 身上：** FlyingFood 在 flying 状态下应可被蛇或 BOSS 吃掉（检测蛇/BOSS 身体与 FlyingFood 坐标重叠）。
6. **蛇在 BOSS 战中全部身体被消耗，只剩头/缩为0：** 蛇长度归零 → gameover → 读取存档 → 存档点在 BOSS 房间之前 → 玩家需要重新到达 GOAL 房间。
7. **BOSS 战中玩家暂停：** 暂停时 BOSS 也应暂停，防止意外死亡。
8. **多个 FlyingFood 同时存在：** 需要设置最大同时存在数量（建议 10 个），避免性能问题。
9. **FlyingFood 落定在障碍物上：** 落定时如果目标格子不是 FLOOR→寻找最近的 FLOOR 格放置。
10. **BOSS 战中房间内的原有食物：** GOAL 房间内事先放置的基础食物在 BOSS 战中保留，作为初始补给。

### Failure Paths

1. **BOSS 战进行中玩家退出/刷新：** 游戏未保存 BOSS 战进度，重新加载后需重新到达 GOAL 房间。
2. **FlyingFood 物理模拟性能问题：** 大量 FlyingFood 同时存在（10+）时的碰撞检测。设置上限（10-15 个），超出则直接落定最早的。
3. **BOSS 子弹速度 > 玩家反应速度：** BOSS 子弹应设置低于玩家子弹的速度（如 projectileSpeed = 1），让玩家有足够时间闪避。
4. **入口关闭致使蛇被夹在门间：** 蛇头刚进入房间但蛇尾仍在门外时关闭入口 → 蛇尾应该被截断（视为门关闭时剪断）或在蛇全部进入后再关闭（延迟 1 tick）。

> 这些直接成为 Plan 阶段的测试用例骨架。

## 7. Spike / Experiment

### Question to Answer

FlyingFood 的物理弹飞模拟在纯 JS 中性能如何？在 20×20 房间内需要多少计算量？

### Method

1. 实现 `FlyingFood` 数据结构 + tick 更新函数（不含渲染）：
   - 10 个 FlyingFood 同时运动
   - 每个带随机初始速度、摩擦力系数 0.95
   - 检测墙壁碰撞并反弹（坐标边界检测）
   - 检测蛇头/BOSS 头重叠
   - 运行 100 tick 模拟
2. 统计：每 tick 的平均操作数、FlyingFood 落定平均时间

### Result

（本 spike 在 Plan 阶段实际运行。预期结论：）

| FlyingFood 数 | 每 tick 操作 | 碰撞检测 (vs snake/boss) | 总耗时 |
|--------------|-------------|------------------------|--------|
| 5            | 15 次更新   | 25 次坐标比较          | < 0.1ms |
| 10           | 30 次更新   | 50 次坐标比较          | < 0.1ms |
| 20           | 60 次更新   | 100 次坐标比较         | < 0.3ms |

每次 FlyingFood 更新仅涉及：位置增量 `vx * 0.95` + 墙壁碰撞边界检查 + 蛇/BOSS 碰撞检测。即使 20 个同时活动，耗时也远低于 1ms。**无需性能优化。**

### Impact on Approach

如果 spike 确认性能可忽略（预期），Approach A 的 FlyingFood 系统落地无风险。如果发现碰撞检测开销过大，可改为仅检测蛇头/BOSS 头位置而非全身体段。
